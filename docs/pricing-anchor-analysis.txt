Important reading rule

- In practice there are 6 anchor families:
  1. Whole-property base anchors
  2. SQM curve anchors
  3. Room-level anchors
  4. Item-level anchors
  5. Scope floors / minimum floors
  6. Rate anchors ($/sqm, $/lm, per-item) with modifiers

============================================================
1. System summary
============================================================

The pricing system is hybrid, not purely anchor-based.

At a high level:
- Apartment entire-property interior pricing is driven by APARTMENT_SQM_CURVE.
- Apartment specific-area interior pricing is driven by APARTMENT_ANCHORS_OIL plus room scoring and room anchors.
- House interior pricing is driven by HOUSE_INTERIOR_ANCHORS plus whole-house bands, room ratios, and a special 3B2B fair calibration curve.
- Exterior pricing is driven by a mix of:
  - wall area rate curves,
  - wall-type scope floors,
  - per-area anchors for non-wall exterior elements,
  - per-item trim anchors,
  - roof/deck/paving rate tables,
  - condition/storey/complexity modifiers.

This means two things:
- Some anchors are literal base prices.
- Some anchors are only guardrails, calibration points, or fallback minimums.

2. Core global pricing controls

2.1 MAX_PRICE_CAP
- Value: 35000
- Role:
  - Global hard cap on interior, exterior, and combined totals.
  - If a result reaches the cap, the UI converts to "From AUD X+ (Site Inspection Required)" style output.
- Effect:
  - Prevents extreme outputs.
  - Also compresses the top end for large premium jobs.
- Likely creation logic:
  - Business safeguard first, market calibration second.
  - Checklist B.16 notes this cap may be too low for very large 3-storey premium jobs.

2.2 STORY_MODIFIER
- 1 storey = 1.0
- 2 storey = 1.18
- 3 storey = 1.35
- Also contains legacy aliases: "Single story", "Double story or more"
- Role:
  - Applied to interior estimates after the base range is assembled.
  - Exterior uses storey mostly through separate storey-aware anchors / rates rather than this exact multiplier.
- Likely creation logic:
  - Empirical access/labour uplift percentages.
  - Chosen to be simple and monotonic.

2.3 Condition multipliers

  - Fair {1.08, 1.22}
  - Poor {1.22, 1.55}
- House interior:
  - Excellent {1.00, 1.08}
  - Fair {1.10, 1.24}
  - Poor {1.22, 1.40}
- Exterior:
  - Excellent {1.00, 1.08}
  - Fair {1.10, 1.24}
  - Poor {1.22, 1.40}
- Role:
  - Expand or shift min/max ranges based on condition/prep intensity.
- Likely creation logic:
  - Apartment model allows stronger discount and stronger poor-condition penalty.

- Important note:
  - Checklist B.5 explicitly flags these three multiplier systems as inconsistent by design or evolution.

2.4 capRangeWidthSmart
- Role:
  - Narrows final range width after calculation.
- Exterior cap policy:
  - min <= 3000 => width cap 1000
  - min <= 8000 => width cap 1800
  - min <= 10000 => width cap 1750

- Interior/total cap policy:
  - min <= 5000 => width cap 1200
  - min <= 10000 => width cap 1800
  - min <= 18000 => width cap 2500
  - else => width cap 3500
- Important note:
  - This is not an anchor, but it strongly shapes the visible output.
  - Checklist B.6 notes this can hide uncertainty, especially for poor-condition jobs.

4.1 HOUSE_INTERIOR_ANCHORS
- Values:
  - 2B1B {7500, 11000, 9000}
  - 3B2B {9500, 14000, 11500}
  - 4B2B {13000, 18500, 15500}
  - 5B3B {17000, 25000, 20000}
- Where used:
  - House + Entire property
  - House + Specific areas only
  - inferHouseKey chooses the anchor bucket.
- How applied for Entire property:
  1. Infer house class from bedroomCount, bathroomCount, or approxSize fallback
  2. Take median as base
  3. Multiply by areaFactor
  4. Apply ENTIRE_HOUSE_BAND by condition
  5. Floor against anchor.min * areaFactor * 0.98
  6. Ceiling against anchor.max * areaFactor * 1.02
  7. Apply storey modifier, trim premiums, complexity, calibration, ceiling type, etc.
- How applied for Specific areas:
  1. Infer house class
  2. Convert room scores to partialRatio
  3. Use anchor.median * partialRatio as baseMid
  4. Convert to 0.86 / 1.24 spread
  5. Apply pseudo-sqm uplift for larger room sets
  6. Enforce hard floor
  7. Apply house condition multiplier
- How it appears to have been created:
  - Class buckets reflect conventional sales logic:
    - small house
    - standard family house
    - large family house
    - premium large house
  - The median values look hand-calibrated to Northern Beaches 2026 expectations.
  - The range widths are not percentage-perfect; they look business-chosen for explainability and realism.

6.1 EXTERIOR_WALL_TYPE_FLOORS
- cladding:

     - full exterior
     - full exterior double/triple
  3. Force rangeMin to be at least the scope floor
  4. Force rangeMax to be at least floor * 1.25
- Why this anchor exists:
  - Protect against underpricing when area/rate logic yields too low a number.
- How it appears to have been created:
  - Market minimum acceptable quote by wall type and scope package.
  - These are not median anchors; they are commercial floor anchors.

6.2 EXTERIOR_WALL_RATE_CURVES
- Each wall type has wallArea checkpoints with min/max $/sqm.
- Example cladding:
  - 60sqm => 28 to 36
  - 95sqm => 25.5 to 33
  - 130sqm => 23.5 to 30
  - 175sqm => 22 to 28
  - 230sqm => 20.5 to 26
  - 320sqm => 19 to 24
- Rendered and brick are consistently higher.
- Where used:
  - Wall pricing path in calculateExteriorEstimate().
- How applied:
  1. Convert floor sqm to footprint sqm using storey count
  2. Estimate wall area = 4.2 * sqrt(sqm) * wallHeight
  3. Interpolate wall min/max rate from curve
  4. wallArea * rate => raw wall component
- Why this anchor exists:
  - Exterior wall work scales more smoothly as a rate curve than as a class anchor.
  - Larger jobs get lower unit rates.
- How it appears to have been created:
  - Hybrid of:
    - geometric wall-area estimation,
    - market $/sqm rates by substrate,
    - volume discount curve.
  - Brick highest, rendered mid, cladding lowest reflects prep/coating complexity.

============================================================
8. How these anchors were likely created
============================================================

This section is partly direct evidence and partly inference from the codebase.

Direct evidence from code/comments
- Multiple files explicitly say the values are calibrated to Sydney Northern Beaches market, 2026.
- Several comments call out:
  - "key calibration point"
  - "updated"
  - "market range"
  - "full exterior scope"
  - "premium market"
- The test suite validates many anchor values against market sanity expectations instead of purely mathematical properties.

Strong inference
- The anchors appear to be hand-calibrated commercial pricing tables, not ML-generated.
- The design style suggests this workflow:
  1. Pick representative market jobs
  2. Decide baseline acceptable ranges
  3. Encode those as medians, min/max bands, floors, or rate points
  4. Add modifiers for condition/storey/access/finish
  5. Add special corrections where the generic model was too inaccurate

Evidence for hand-calibration
- The 85sqm apartment point is explicitly singled out.
- 3B2B fair single-storey house gets its own custom interpolation curve.
- Exterior has both rate curves and package floors, which is typical when a pure formula underquotes common jobs.
- Trim/front door pricing is separated because bundled pricing was probably too noisy or inaccurate.

Why some anchors are min/max/median and others are rates
- Whole-property jobs:
  - Better handled by anchor/median models because customers think in property package terms.
- Partial rooms and trim:
  - Better handled by room or item anchors because scope is discrete.
- Exterior walls, roofs, deck, paving:
  - Better handled by rate logic because size materially changes labour and materials.
- Floors exist because rate logic alone tends to underquote small or awkward jobs.

============================================================
9. Current inconsistencies and maintenance risks
============================================================

These are important if someone wants to trust or extend the anchor list.

9.1 Apartment has two baseline systems
- Entire property uses APARTMENT_SQM_CURVE.
- Specific areas uses APARTMENT_ANCHORS_OIL.
- Checklist B.8 documents this intentional split.

9.2 Some important pricing bands live outside pricing-engine.ts
- ENTIRE_HOUSE_BAND and ENTIRE_APT_BAND are in generate-painting-estimate.ts.
- Checklist B.9 flags this as a single-source-of-truth issue.

9.3 Generic Exterior Trim anchor is conceptually present but operationally weak
- Engine still contains it.
- Product policy currently requires detailed trim item selection.
- So maintainers should treat per-item trim anchors as the real production path.

9.4 Deck and paving models are different in shape
- Deck uses base rates plus separate area multiplier.
- Paving uses area-banded rates.
- This raises comparison/maintenance complexity.

9.5 Range caps can hide true anchor spread
- Final displayed range may be much tighter than raw anchor-derived range.
- If someone audits anchors only from UI output, they may misread the underlying calibration.

12. Bottom line
============================================================

The app does not have one single "price anchor list".
It has a layered pricing architecture where:
- medians anchor whole-property expectations,
- room and item anchors break that expectation into partial scopes,
- rate tables price scale-sensitive work,
- floors protect minimum commercial viability,
- modifiers and caps shape the final customer-visible range.

The anchors were almost certainly created by manual market calibration against Sydney Northern Beaches 2026 expectations, then refined through edge-case fixes and regression tests.

If this system is expanded further, the biggest long-term cleanup opportunity is to:
- centralize all anchor/band definitions in pricing-engine.ts,
- decide whether apartment pricing should be curve-led or class-led,
- and remove any remaining legacy fallback anchors that are no longer part of the intended product behavior.
